## Phase 1 ‚Äî Data layer

**Status:** done ‚Äî landed on `cursor/phase-1-data-layer-2069`.

**Goal:** the merged `Controller[]` is observable in‚Äëapp, with live prices from Shopify and the full per‚Äëdevice spec set from a hand‚Äëcurated file.

1. ‚úÖ `src/types/controller.ts` ‚Äî `SpecValue` discriminated union, `Controller` with open `specs: Record<string, SpecValue>`, `SpecCatalog`/`ControllerSpecEntry` typed separately.
2. ‚úÖ `src/data/specCatalog.ts` ‚Äî canonical labels, sections, `booleanByDefault`, display order, raw ‚Üí canonical normaliser.
3. ‚úÖ `src/data/controllerSpecs.json` ‚Äî entries for the 8 in‚Äëscope controllers. **Important correction during implementation:** the planned handles were guesses; live Shopify slugs differ in 5 of 8 cases (e.g. `8bitdo-pro-3-bluetooth-gamepad` is actually `8bitdo-pro-3`; `8bitdo-sn30-pro-hall-effect-joystick` is actually `8bitdo-sn30-pro-bluetooth-gamepad`). Real handles now match the live store.
4. ‚úÖ `src/services/shopify.ts` ‚Äî `fetchControllerProducts()` paging `/products.json` until empty, filtering to `product_type === "Game Controllers"`, normalising to `{ shopifyHandle, storeTitle, imageUrl, shopUrl, basePriceUSD, compareAtPriceUSD, onSale, available }`. Price‚Äëminimum follows store‚Äëcard semantics: lowest available variant, fall back to lowest unavailable when nothing is in stock. `fetch` is dependency‚Äëinjectable for tests.
5. ‚úÖ `src/services/catalog/` ‚Äî split for clean dependencies:
   - `merge.ts` (pure `mergeOne` / `mergeCatalog` / `curatedSpecsFromRaw`, no module‚Äëload side effects)
   - `cache.ts` (`localStorage` SWR with injectable `Storage`)
   - `index.ts` (`loadCatalog()` and `loadCatalogSWR()` async‚Äëgenerator that yields cached ‚Üí fresh)
6. ‚úÖ `src/data/fallbackCatalog.json` regenerated by `scripts/refresh-fallback.ts` (`pnpm refresh-fallback`). Currently snapshots all 39 Shopify products (the 8 curated ones get full specs, the rest come back `specsPending: true` so the grid can label them).
7. ‚úÖ Tests ‚Äî 46 in 6 files:
   - `shopify.test.ts` ‚Äî price‚Äëminimum across 6 cases (lowest available, sale detection, all unavailable, empty, equal compare, etc.); image fallback; pagination loop and error handling.
   - `catalog/merge.test.ts` ‚Äî spec‚Äëfound, spec‚Äëmissing, missing‚Äëshopify, name synthesis.
   - `catalog/cache.test.ts` ‚Äî write/read roundtrip, malformed JSON, quota errors, TTL boundary.
   - `catalog/index.test.ts` ‚Äî fresh load, persistence, fetch‚Äëfail fallback (cache + bundled), SWR stale/fresh paths.
   - `data/specCatalog.test.ts` ‚Äî label normalisation, schema invariants between catalog and raw map.
   - `App.test.tsx` ‚Äî heading smoke test.

**Exit criterion (met):** App renders a JSON‚Äëbacked table of merged Shopify pricing √ó curated specs with correct prices, sale detection, and `specsPending` badges. All four CI gates pass locally.

**Heads-up for Phase 2:** `Date.now()` cannot be called during render under React 19's hooks plugin (`react-hooks/purity` rule). For relative timestamps, capture `Date.now()` inside the effect that sets state and store it next to the snapshot.

## Phase 2 ‚Äî Browse view

**Goal:** the user can see, filter, and select controllers.

1. `CatalogContext` and `CatalogProvider` exposing `controllers`, `status`, `fetchedAt`, `refresh()`. Provider sits at the app root.
2. `CompareContext` and `CompareProvider`:
   - Holds `selectedIds: string[]` (cap 3), with `add(id)`, `remove(id)`, `clear()`, `toggle(id)`.
   - `toggle` is a no‚Äëop when at cap and the id isn't already selected.
   - Persists to `localStorage` under `compare:v1`.
3. `ControllerCard` component ‚Äî image, name, family chip, price (with strike‚Äëthrough compare‚Äëat if on sale), spec icons (connectivity badges, Hall‚ÄëEffect/TMR badge if applicable, Switch/PC/Mac glyphs), and the "+ Compare" / "‚úì In comparison" / "Cap reached" button states.
4. `FiltersSidebar` component ‚Äî compatibility multi‚Äëselect, connectivity multi‚Äëselect, joystick‚Äëtype select, price‚Äërange slider, on‚Äësale toggle, text search. Filters live in component state and produce a filter predicate composed in a single memoised selector.
5. `CompareBar` component ‚Äî fixed bottom bar visible on Browse when 1+ controllers are selected; shows chips with name + remove ‚úï, and a "Compare (N)" CTA that routes to `/compare?ids=‚Ä¶`.
6. Wire the Browse page (`/`) to: `CatalogProvider` data + `FiltersSidebar` + grid of `ControllerCard` + `CompareBar`.
7. Tests:
   - Filter predicate unit tests (each filter dimension independently and combined).
   - `CompareContext` reducer tests (add at cap is a no‚Äëop; remove; persist round‚Äëtrip).
   - One RTL test that renders Browse with stubbed catalog, selects two controllers, asserts the CompareBar count.

**Exit criterion:** I can browse, filter, and accumulate up to 3 controllers; the bottom bar reflects the selection across refreshes.

## Phase 3 ‚Äî Compare view

**Goal:** the selected controllers render as a side‚Äëby‚Äëside comparison with differences highlighted.

1. `/compare` route reads `ids` from the query string, resolves to `Controller[]`, drops missing ones with a toast.
2. `ComparisonGrid` component ‚Äî built with CSS Grid on `<div>` elements (no `<table>`, see design ¬ß11). Rows are derived from the union of all `specs` keys across the selected controllers, then grouped by the section assigned in `specCatalog.ts` (`Pricing & availability`, `Connectivity`, `Compatibility`, `Sticks & triggers`, `Buttons & feedback`, `Battery & physical`, `Software`, `Other`). Each row is a `{ canonicalLabel, values: SpecValue[] }` where `values[i]` is the spec value for the `i`th selected controller (or a `missing` marker if that controller does not list this spec). The renderer is per‚Äë`SpecValue.kind` (price formatter for pricing rows, list‚Äëof‚Äëpill renderer for `list`, boolean check/‚úï for `boolean`, key:value table for `perPlatform`, etc).
3. Three‚Äëstate difference highlighting (per design ¬ß7):
   - **All equal** ‚Üí quiet row.
   - **All present but values differ** ‚Üí row accent + emphasised cell background + non‚Äëcolour dot indicator on differing cells.
   - **Present on some, missing on others** ‚Üí stronger row accent; missing cells render `‚Äî` with a tooltip. A small `classifyRow(values, catalogEntry)` helper returns one of `"equal" | "differ" | "partial"`.
     Section groups that end up with zero rows after filtering (none of the selected controllers expose any spec in that section) are collapsed.
4. Column header ‚Äî image, name, sale badge, link to the official product page, "‚úï Remove" button that updates `CompareContext` and the URL.
5. Empty‚Äëcolumn slot when fewer than 3 are selected ‚Äî a "+ Add controller" cell that links back to `/`.
6. Responsive: the grid template collapses from `minmax(140px, auto) repeat(N, 1fr)` on `‚â• md` to `1fr` on `< md`. On mobile each controller becomes a vertical card with the row label shown inline next to each value (e.g. "Connectivity: Bluetooth, 2.4G"). Same DOM, different `grid-template-columns` and a couple of `display` swaps ‚Äî no conditional rendering.
7. Tests:
   - `classifyRow` unit tests across each `SpecValue.kind` and all three classification outcomes (equal / differ / partial), including array order normalisation for `list` and key‚Äëorder normalisation for `perPlatform`.
   - Row‚Äëderivation unit test: union of spec keys across 3 controllers produces the expected ordered row list per section.
   - RTL test: render with 3 fixture controllers ‚Äî assert that the price row has the differ accent, the equal Vibration row is quiet, and an "RGB Fire Ring" row present on only one controller renders the partial accent + `‚Äî` cells for the other two.

**Exit criterion:** dropping 2‚Äì3 fixtures into the URL produces a readable comparison; removing a column updates URL + state in sync.

## Phase 4 ‚Äî Polish, a11y, deploy

**Goal:** the app feels intentional and ships.

1. Accessibility pass: semantic `<section>` + heading structure (no `<table>`), focus rings, keyboard navigation of the compare bar, `aria-live="polite"` toast region, colour‚Äëindependent diff indicator (small dot/icon).
2. Empty/loading/error states: skeleton cards on first load, banner for "live pricing unavailable ‚Äî showing cached prices from <time>", toast for dropped URL ids, "no results" state for filters.
3. Lighthouse pass on `pnpm preview`; fix anything below 90 / 95 perf / a11y.
4. Vercel deployment: add `vercel.json` if needed (likely not ‚Äî Vite default works), connect repo, verify preview URLs on PRs.
5. README updates: live link, screenshot, contribution guide for adding a new controller to the spec file.

**Exit criterion:** the deployed URL loads in < 2s on a fresh visit, scores ‚â• 90/95 on Lighthouse, and one fresh visitor can complete the primary flow without prompting.

## Risks & how each is addressed

| Risk                                                                            | Mitigation                                                                                                                                                                                |
| ------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 8BitDo locks `products.json` behind CORS                                        | Add a tiny Vercel Edge Function proxy; the rest of the app does not change because `shopify.ts` is the only caller.                                                                       |
| Spec file drifts as 8BitDo releases new controllers                             | `specsPending: true` path keeps the app functional; CI can grow a daily job that diffs Shopify's catalog against the spec file and opens an issue. Out of scope for v1, but cheap to add. |
| "Base price" misrepresents bundles where the cheapest variant is a stripped SKU | We display the same number the store card shows. If this is ever misleading for a specific controller, add a `priceFloorOverride` field to the spec entry.                                |
| Shopify pagination shape changes                                                | The fetcher pages until an empty response and is the only contact point with Shopify; one focused unit test pins the shape we depend on.                                                  |
| Project bloats with feature requests                                            | Non‚Äëgoals in the design doc are explicit. New asks land as separate spec docs, not as scope creep on v1.                                                                                  |

## What I did NOT include and why

- **Server‚Äëside rendering / Next.js.** No SEO benefit (the app is for an individual decision, not discovery), no auth, no per‚Äërequest data. The cost of an SSR runtime would not be repaid.
- **A component library (shadcn, MUI, Mantine).** Two screens, ~6 components total; utility CSS is faster and leaves no abandoned dependencies.
- **A backend / database.** All data is either public (Shopify) or static (spec file). Adding a backend is the kind of decision that's easy to add later and hard to remove.
- **State libraries (Redux, Zustand, Jotai).** Two contexts cover the entire state surface; reaching for a library here would be ceremony.
- **An LLM extraction pipeline for specs.** ~16 SKUs; manual curation is faster than building and reviewing an extraction.
